# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

A more detailed backend-specific guide lives at `backend/CLAUDE.md` — read it when touching FastAPI routes, models, the license verifier, the lab module, or the PDF/print pipeline.

## Product

**KT HEALTH ERP** — Hospital management system for local-network deployment. Ships either as a Windows `.exe` bundle (PyInstaller) or runs from source. Distribution is gated by Ed25519-signed `.lic` files generated by the separate License Manager app. Sales can be direct or via 3rd-party vendors ("sellers") whose contact info is embedded in the signed license.

## Repository Layout

This is a multi-app monorepo, not a workspace — each app has its own dependencies and lifecycle.

```
backend/            FastAPI server (port 8000) — the hospital app
frontend/           React 18 + Tailwind + shadcn/ui (port 3000) — the hospital UI
license-manager/    Separate FastAPI + React app (port 9000) for issuing .lic files
license-tools/      Standalone Ed25519 keypair + license generator (CLI)
*.py at root        install_and_setup.py, run_backend.py, run_frontend.py — convenience launchers
build_exe.bat       Windows packaging entry point (frontend build → pip install → pyinstaller)
TODO_*.md           Per-feature planning docs (historical, may be stale)
```

## Common Commands

### First-time setup
```bash
python3 install_and_setup.py          # Creates backend venv, installs deps, sets up DB
```

### Run the hospital app (two terminals)
```bash
# Terminal 1 — backend on :8000 (proxied by frontend)
cd backend && ./venv/bin/python main.py
# or from root: python3 run_backend.py  (wraps backend/start_server.py)

# Terminal 2 — frontend dev server on :3000
cd frontend && npm start
# or from root: python3 run_frontend.py
```

### Run the License Manager (separate app, do not confuse with the hospital app)
```bash
cd license-manager/backend && python app.py    # :9000
cd license-manager/frontend && npm start
```

### Tests
Only one pytest suite exists today: an end-to-end inpatient smoke test that overrides the DB dependency with a temp SQLite file (see `backend/tests/conftest.py`).
```bash
cd backend && ./venv/bin/python -m pytest tests/                  # all tests
cd backend && ./venv/bin/python -m pytest tests/test_inpatient_smoke.py::TestInpatientE2E::test_create_room -v
```
There is no frontend test suite wired up despite `react-scripts test` being available.

### Migrations
SQLAlchemy `create_all()` runs at backend startup (creates missing tables). Column additions are handled by idempotent migration scripts that run automatically on startup — `migrate_patient_fields.py` is invoked from `main.py`'s `startup_event`. Other `migrate_*.py` scripts in `backend/` are one-shot helpers run manually if needed. To add a column, append to the `NEW_COLUMNS` list in `migrate_patient_fields.py` rather than writing a new migration file.

### Windows packaging
```bash
build_exe.bat        # MUST be used — running PyInstaller directly bundles a stale frontend build
```
Output is `backend/dist/KTHEALTHERP.exe`. Bundled mode persists data to a `data/` folder next to the .exe; path resolution lives in `backend/app/utils/paths.py` (`get_db_path`, `get_uploads_dir`, `get_frontend_dir` all check `sys.frozen`).

## High-Level Architecture

### Three apps, one repo
1. **Hospital app** (`backend/` + `frontend/`) — what end-users run. Single FastAPI process serves the API on `:8000` and, in bundled mode, also serves the built React assets. Frontend in dev mode proxies API calls via `"proxy": "http://localhost:8000"` in `frontend/package.json`.
2. **License Manager** (`license-manager/`) — vendor-facing app for issuing `.lic` files. Signs payloads with an Ed25519 private key. The hospital app verifies with the matching public key. These keys also exist in `license-tools/keys/` for CLI generation.
3. **License Tools** (`license-tools/`) — minimal CLI used historically before the License Manager existed. Produces the same `.lic` format.

### Hospital app — the four cross-cutting systems
These four systems shape how every feature behaves; understand them before adding routes or pages.

**1. Auth + multi-role permissions.** JWT (HS256, 24h), bcrypt passwords. Users have a primary `role_id` plus many-to-many roles via `user_role_association`. Permission checks chain: `super_admin`/`hospital_admin` bypass → module enabled → license includes feature → role-module permission. Two decorators: `require_permission(module, action)` (legacy action-bucket check, still used by lab/pharmacy/outpatient) and `require_feature_permission(module, permission_name)` (granular per-feature check, used by all 141 inpatient routes). The inpatient module defines ~54 granular permission keys and a per-role default matrix — see `backend/CLAUDE.md` for the full vocabulary and matrix. Hospital admins can customise grants via Hospital Administration → Role Permissions.

**2. Module gating.** A module is "enabled" only when BOTH the admin toggle is on AND the loaded license's `features` array includes it. The frontend reads `/api/system/enabled-modules` to decide nav visibility and route access. Toggling a module the license doesn't include is rejected by the backend (`PUT /api/admin/modules/{id}`). Uploading a new license auto-disables modules dropped from `features`.

**3. License lifecycle.** `.lic` files are base64 JSON + Ed25519 signature, bound to a machine ID. `app/services/license_service.py` verifies signature + machine ID and stores the parsed payload in the DB. The signed payload includes `seller_info` (vendor name/address/phone) which auto-populates the PDF footer ("Powered by KT HEALTH ERP — Sold by {vendor}") and the in-app Support Contact page.

**4. PDF/print pipeline.** All printable output (bills, prescriptions, lab reports, discharge summaries) goes through `backend/app/utils/pdf_service.py` (ReportLab) with a uniform `include_header` toggle. Frontend uses one shared utility — `frontend/src/utils/printPdf.js` — that fetches the PDF as a blob, opens a preview dialog with an iframe, exposes a "Include header" checkbox that re-fetches, and prints via a hidden `iframe.contentWindow.print()`. Preserve this pattern when adding new printable artifacts; do not invent a parallel print path.

### Backend conventions worth knowing up front
- Routes live in `backend/app/routes/` (one file per domain, each is a router with a prefix); models in `backend/app/models/`; cross-cutting helpers in `backend/app/utils/`; service-layer business logic in `backend/app/services/`.
- DB engine is wrapped in a lazy `_EngineProxy` (`config/database.py`) so the setup wizard can change the DB path at runtime without restarting.
- Audit logging is hybrid: `AuditMiddleware` records request-level events automatically, but routes also call `log_action()` explicitly for domain events.
- Patient identity is split: some endpoints take the UUID `patient_id`, others take the integer `id`. Check the endpoint signature.
- Pydantic 422 errors return an array of objects, not a string. Frontend code must guard with `typeof err.detail === 'string'` before rendering — there are existing utilities for this.
- API URLs in the frontend are always relative (no hardcoded host) so the bundled .exe works on any LAN IP.

### Frontend conventions
- React 18 + functional components, Tailwind + shadcn/ui (Radix primitives), `@tanstack/react-query` for server state, `react-hook-form` for forms, `lucide-react` for icons. (Note: README mentions Material UI — that is outdated; the actual stack is shadcn/Tailwind.)
- Routing: `App.js` → `Dashboard.js` is the central layout that owns the sidebar nav and renders role-based routes inline. Module visibility flows from `enabledModules` (loaded from the backend) through `Dashboard.js`.
- Auth state lives in `contexts/AuthContext.js` — token + user + licenseStatus stored in `localStorage`, with a global axios 401 interceptor that logs out on token expiry.
- Module pages live in `frontend/src/pages/modules/`; reception sub-pages live in `frontend/src/pages/modules/reception/`.

### Installation / setup wizard
**There is no React Setup Wizard anymore** — the route (`/api/setup/*`) and the frontend component were removed when the Inno Setup installer became the single install path. Any older mention of `SetupWizard.js` or `/api/setup/complete` in docs is stale. The only first-install flow is:

1. Windows operator runs `installer/installer.iss` (built into `KTHEALTHERP_Setup_<ver>.exe`). The wizard collects: data folder mode (fresh / adopt-existing / restore-backup), optional license, hospital details, admin account, backup folders.
2. On the wizard's last step (`ssPostInstall`), `WriteSeedFile` (installer.iss:1016) writes `{app}\data\install_seed.json` plus, for fresh installs, `{app}\data\.install_seed.pwd` (ACL-locked to SYSTEM + Administrators).
3. `backend/launcher.py:282` calls `app.services.bootstrap_from_seed.consume_seed_if_present(exe_dir)` on first launch. That function dispatches by `mode`:
   - `fresh` → `init_database_and_seed` (db_seed.py) creates the hospital, super_admin, roles, modules, then `store_license` if a `.lic` was selected.
   - `adopt_existing` → just rebinds `config.json` + the engine to the existing DB.
   - `restore_backup` → SQLite `.backup()` from the chosen `.db` into the target folder, then runs `migrate_patient_fields` to bring schema forward.
4. On success it deletes the two seed files. On failure it writes `data/.bootstrap_status.json` and **leaves the seed in place** so a retry is possible.

**Pre-install validation is performed by `installer/bin/dbcheck.exe`** (PyInstaller-built from `installer/dbcheck/dbcheck.py`). The Inno Setup Pascal code shells out to it for: `machine-id`, `check-db`, `check-writable`, `validate-license`, `validate-backup-db`. Each command writes a single JSON line.

**Source-mode (macOS / Linux dev) has no install flow.** `python main.py` with no `config.json` simply prints "Setup not complete" and exits the startup work — `consume_seed_if_present` is **only** called from `launcher.py` (the Windows-bundled launcher). To bootstrap a dev DB you either need to hand-write `install_seed.json` and drop it into `data/`, or call `db_seed.init_database_and_seed(seed, db_path)` directly from a Python shell.

**dbcheck.exe is built with `sqlalchemy` excluded** (`dbcheck.spec:42`). Anything it imports from the backend must therefore avoid `sqlalchemy.orm` at module-import time. `app.services.license_service` currently violates that (top-level `from sqlalchemy.orm import Session`), so `validate-license` raises `ModuleNotFoundError: No module named 'sqlalchemy'` and the wizard surfaces "License rejected: No module named sqlalchemy". Fix is to keep pure-verification helpers (`inspect_license_file`, `verify_license_machine_binding`, `compute_license_status`) in a Session-free module that both license_service and dbcheck can import.

After the wizard, the hospital admin can upload or replace a `.lic` via Dashboard → License Management.

### Backup
SQLite `.backup()` API (not `shutil.copy2`) for safe online copies. A background thread mirrors to configured destination(s) every 60 seconds. Configuration lives in `backend/config.json`.

## Important Notes

- **Brand vs display name**: `KTHEALTHERP` (brand, used in filenames/exe), `KT HEALTH ERP` (display, used in UI strings). README still says "Hospital ERP System" — that is outdated branding.
- **Two databases visible in `backend/`**: `hospital_erp.db` (legacy) and `kthealth_erp.db` (current). The current code uses `kthealth_erp.db`; the legacy file is kept for old installations.
- **No default credentials are shipped** — the very first super_admin is created by the Inno Setup wizard (fresh mode) from the operator-typed username + password. `setup_initial_data.py` and any other "default admin/admin123" docs are historical and no longer wired in.
- **CORS is `*` and the server binds `0.0.0.0`** — intentional for LAN access. Do not "harden" this without understanding the deployment model.
- **TODO_*.md files at the repo root are historical planning notes**, not living documentation. Do not treat them as authoritative for current behavior — read the code.
